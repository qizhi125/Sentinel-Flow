# 数据生命周期 (Data Lifecycle)

本文档描述了一个数据包在 Sentinel-Flow v6.0 中从网卡到屏幕的“奇幻漂流”。

## 第一阶段：入站 (内核态 -> 用户态)
1.  **抵达**：数据包到达网卡接收队列。
2.  **打戳**：Linux 内核打上纳秒级时间戳 (`kernelTimestampNs`)。
3.  **拷贝**：`PcapDriver` 从 **对象池** 中借出一个空对象，将数据拷贝进去。
4.  **哈希**：驱动计算 `Hash(源IP, 目的IP, 协议)` 以确定 `WorkerID`。
5.  **分发**：数据包被推入 `WorkerQueue[WorkerID]`。

## 第二阶段：处理 (流水线)
1.  **获取**：`WorkerThread` 从队列弹出数据包。
2.  **分诊**：
    * 如果是 `SYN/FIN/RST` -> 插队优先处理。
    * 如果是 `Payload > 1500` 且 `系统拥塞` -> 直接丢弃 (智能卸载)。
3.  **分析**：
    * **Parser**：解析头部信息。
    * **Session**：更新连接状态表。
    * **Inspector**：进行正则匹配和特征检测。
4.  **产出**：生成一个轻量级的 `AnalysisResult` 结构体（不再持有原始数据）。

## 第三阶段：出站 (引擎 -> UI)
1.  **聚合**：结果被写入线程局部的聚合缓冲区。
2.  **推送**：每隔 200ms，缓冲区被交换（Swap）给 **表现层**。
3.  **回收**：原始的 `RawPacket` 对象被“归还”给对象池（Reset）。
4.  **渲染**：UI 线程读取聚合数据，绘制图表。

## ⚠️ 核心原则：时间戳真实性
我们 **绝不** 使用 `QDateTime::currentDateTime()` 来显示数据包时间。
无论处理延迟多少，我们始终透传 `kernelTimestampNs`，确保取证的法律效力。
